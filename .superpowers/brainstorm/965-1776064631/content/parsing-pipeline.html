Section 3 — Parsing Pipeline

How source files become lineage graphs — one pipeline, three parsers

Parsing Pipeline Flow
STAGE 1
File Ingestion
Git clone · API pull · ZIP extract → flat file list
→
OUTPUT
FileRecord{ path, content, type }
type: "notebook" | "python" | "sql"
▼ route by file type
.ipynb Parser
lib: nbformat
1. Extract code cells
2. Detect cell language
   (SQL magic / PySpark)
3. Route to SQL or
   PySpark sub-parser
4. Tag with cell index
.py PySpark Parser
lib: Python ast
1. Parse AST
2. Find DataFrame ops:
   read, select, withColumn,
   groupBy, agg, join, filter
3. Resolve column exprs
4. Track df variable aliases
.sql Parser
lib: SQLGlot
1. Parse to AST
2. Walk SELECT exprs
3. Resolve CTEs,
   subqueries, aliases
4. Detect window fns,
   aggregations, casts
5. Map JOIN keys
▼ emit ColumnNode + LineageEdge per expression
STAGE 3 — Lineage Engine (in-memory graph)
Graph Builder
Merge all edges into a
directed acyclic graph.
Nodes = columns,
Edges = transformations.
Upstream Resolver
Walk graph backwards
from a target column
to all root sources.
BFS/DFS traversal.
Impact Analyzer
Walk graph forward
from a source column
to all downstream
dependents.
Search Index
Simple in-memory
inverted index over
table/column names
for fast lookup.
▼ JSON response
STAGE 4 — REST API Endpoints
POST /sources — register a source
GET /tables — list all tables
GET /tables/:id/columns — list columns
GET /lineage?table=&column= — column lineage
GET /impact?table=&column= — downstream impact
GET /search?q= — search columns/tables
